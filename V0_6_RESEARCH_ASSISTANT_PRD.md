# v0.6 Research Assistant PRD

## 1. 文档定位

本文档定义 `Gist / AI4Research` 的 `v0.6 研究助手` 版本产品方案。

`v0.6` 的核心目标是：

- 让 AI 不只服务于单条资料解析
- 让 AI 能围绕“项目”持续工作
- 让 AI 能基于项目上下文、历史条目和用户主动提供的研究背景，做更高层的协作

这一版将把产品从：

- 信息采集工具
- 单条摘要工具
- 条目管理工具

推进到：

- 项目级研究助手
- 多资料对比助手
- 灵感反查与资料联动助手
- 周期性研究回顾助手

---

## 2. 版本目标

### 2.1 核心目标

1. 支持项目级 AI 总结
2. 支持多文献对比
3. 支持基于灵感反查相关资料
4. 支持周回顾 / 未读积压 / 即将截止提醒
5. 支持 Markdown / 周报 / 文献清单导出
6. 新增“研究背景 Markdown 文档”上传与上下文联动能力

### 2.2 非目标

本阶段不追求：

- 完整自动开题报告生成
- 完整学术写作替代
- 云端多人协作编辑
- 向量数据库或复杂知识图谱
- 自动生成高可信研究结论

---

## 3. 核心设计原则

### 3.1 AI 是研究助理，不是研究替代者

系统应该输出：

- 项目总结
- 对比结果
- 相关资料建议
- 可验证假设
- 待补问题

而不是：

- 最终结论
- 已验证创新点
- 可直接发表内容

### 3.2 项目级上下文优先

`v0.6` 里的 AI 不再只基于单条 paper/article 工作。

优先级应为：

1. 用户显式上传的研究背景文档
2. 当前项目知识上下文
3. 当前条目及其关系网络
4. 历史条目库

### 3.3 结构化与可读并存

系统内部优先使用结构化字段和关系数据；
对用户输出时使用可读的 Markdown 和卡片式总结。

### 3.4 人工可控

所有关键 AI 结果应允许：

- 重新生成
- 导出
- 保存为灵感
- 与原始依据一起查看

---

## 4. 用户问题

当前系统已经能采集、解析、整理条目，但仍然存在以下痛点：

1. 项目维度缺乏高层总结
2. 条目之间虽然开始有关系，但用户仍需手动比对
3. 灵感与资料之间的回查链路不够强
4. 用户脑海中的研究方向没有一个稳定入口可以显式输入给系统
5. 周期性研究回顾仍然需要手工完成

因此，需要在 `v0.6` 中引入“研究背景文档 + 项目级 AI 协作”。

## 5. 用户故事

### 5.1 项目型科研用户

我希望打开某个项目时，AI 能告诉我：

- 这个项目最近在研究什么
- 哪些资料最重要
- 当前灵感集中在哪些问题上
- 哪些方向还没有补齐

### 5.2 多文献阅读用户

我希望把两篇或多篇资料放在一起，让 AI 帮我比较：

- 相同点
- 不同点
- 能否组合
- 哪个更适合当前项目

### 5.3 灵感驱动用户

我希望当我写下一条灵感后，系统能反查：

- 相关论文
- 相关 article
- 哪些历史资料可能支持这条想法

### 5.4 研究背景显式输入用户

我希望上传一份 Markdown 文档，描述：

- 最近研究方向
- 当前困惑
- 想解决的问题
- 研究边界和限制

这样系统后续在做分析时，不是“盲分析”，而是结合我的真实研究背景。

## 6. 新增关键能力：研究背景 Markdown 文档

这是 `v0.6` 的新增核心能力之一。

### 6.1 能力定义

用户可以上传一个 Markdown 文档，内容用于描述：

- 当前研究方向
- 最近关注的问题
- 当前项目背景
- 方法偏好
- 数据限制
- 研究目标
- 暂时不考虑的方向

该文档作为长期研究上下文参与后续 AI 分析。

### 6.2 目标

让系统在分析“新资料”或“已有条目”时，能回答：

- 这和用户当前研究方向相关吗
- 这是否能帮助用户解决当前问题
- 这条内容对当前研究边界是否有价值

### 6.3 适用场景

1. 新论文解析后联动分析
2. 旧条目重分析
3. 项目总结
4. 多文献对比
5. 灵感反查相关资料

### 6.4 第一版产品策略

第一版建议：

- 每个项目支持上传 1 份主背景 Markdown 文档
- 允许后续替换
- 不做复杂版本管理
- 不做多人协同编辑

### 6.5 存储建议

建议双存储：

1. 文件存储
- 原始 Markdown 文件落盘

2. 结构化存储
- 保存背景摘要与关键字段

推荐字段：

- `project_context_markdown_path`
- `project_context_summary`
- `project_context_keywords`
- `project_context_last_updated_at`

### 6.6 AI 使用方式

上传后先执行一次背景解析，输出：

- 研究主题
- 当前问题
- 方法偏好
- 数据约束
- 研究目标
- 不关注方向

后续调用 AI 时，不直接把整份 Markdown 每次原样塞进去，而是：

- 优先用解析后的结构化背景摘要
- 必要时拼接原文关键片段

## 7. 功能范围

### 7.1 功能包 A：项目级总结

#### 目标

让用户打开项目时，能看到 AI 生成的“项目状态总结”。

#### 输入

- 项目概览
- 最近新增条目
- 重点论文
- 最近灵感
- 条目关系
- 项目背景 Markdown 文档摘要

#### 输出

- 当前项目主题
- 最近进展
- 重点文献
- 灵感聚焦点
- 待补问题
- 下一步建议

#### 交互

- 项目页新增“AI 项目总结”
- 支持“刷新总结”
- 支持导出为 Markdown

#### 第一版边界

- 先不做历史版本对比
- 先不自动周期性生成

---

### 7.2 功能包 B：多文献对比

#### 目标

支持用户把多条 paper/article 放在一起进行结构化比较。

#### 第一版支持范围

- 先支持 2 篇对比
- 后续扩展 3~5 篇

#### 输入

- 两条或多条选定条目
- 阅读卡
- medium summary
- 关联关系
- 当前项目背景摘要

#### 输出

- 相同点
- 差异点
- 互补点
- 哪些部分可结合
- 适合当前项目的切入点

#### 交互

- 条目详情页新增“加入对比”
- 单独对比结果页

---

### 7.3 功能包 C：基于灵感反查相关资料

#### 目标

用户写下一条灵感后，系统能倒查本地知识库与当前项目上下文。

#### 输入

- 灵感标题 / 正文
- 灵感关联条目
- 项目背景 Markdown 摘要
- 项目内条目库

#### 输出

- 最相关论文
- 最相关资料
- 相关原因
- 建议先读内容
- 可继续追问的问题

#### 交互

- 灵感详情页新增“反查相关资料”
- 可一键建立关联

---

### 7.4 功能包 D：周回顾 / 未读积压 / 即将截止提醒

#### 目标

帮助用户建立研究节奏感。

#### 子能力

1. 周回顾
- 本周新增条目
- 本周重点论文
- 本周灵感变化
- 本周最活跃项目

2. 未读积压
- 未读论文数
- 未读资料数
- 超过若干天未读的重点条目

3. 即将截止提醒
- 比赛截止
- 项目相关提醒

#### 输出形式

- 数据统计卡
- AI 总结卡

#### 第一版策略

- 统计先规则生成
- AI 只负责将统计转成自然语言回顾

---

### 7.5 功能包 E：导出 Markdown / 周报 / 文献清单

#### 目标

让项目内容从 App 内部流出，进入外部写作和汇报流程。

#### 第一版导出类型

1. 项目 Markdown 包
2. 周报 Markdown
3. 文献清单 Markdown

#### 后续可扩展

- CSV
- BibTeX
- PDF

#### 输出目录建议

```text
exports/
  projects/
    {projectName}/
  weekly-reports/
  literature-lists/
```

## 10. 项目背景 Markdown 文档工作流

### 10.1 上传流程

1. 用户在项目页点击“上传研究背景”
2. 选择 `.md` 文件
3. 系统保存原始 Markdown 文件
4. AI 对文档做结构化解析
5. 保存背景摘要
6. 项目页显示“当前研究背景已更新”

### 10.2 使用流程

后续以下能力默认读取该项目背景：

- 项目级总结
- 多文献对比
- 灵感反查
- 新条目联动分析

### 10.3 解析建议

项目背景文档建议提取：

- 核心研究主题
- 当前最关心的问题
- 已尝试的方法
- 不希望偏离的边界
- 数据/资源限制
- 想要的输出或结果

## 13. MVP 建议

如果先做 `v0.6 MVP`，建议范围如下：

1. 上传项目背景 Markdown 文档
2. 项目级总结
3. 灵感反查相关资料
4. 双文献对比
5. 导出 Markdown

先不做：

- 自动提醒
- 自动周期周报
- 超过 2 篇以上复杂对比
- 多格式导出

## 14. 验收标准

### 14.1 项目背景文档

1. 用户可为项目上传 Markdown 文档
2. 系统可保存原始文件
3. 系统可生成背景摘要
4. 项目后续 AI 分析能引用该背景

### 14.2 项目级总结

1. 项目页可生成 AI 总结
2. 总结内容能体现项目内近期变化
3. 总结能参考研究背景文档

### 14.3 多文献对比

1. 用户可选择两条资料进行对比
2. 系统能输出共同点、差异点、互补点
3. 输出能参考项目背景

### 14.4 灵感反查

1. 灵感详情页可触发反查
2. 系统能返回相关论文和资料
3. 返回结果能解释“为什么相关”

### 14.5 导出

1. 项目 Markdown 可导出
2. 周报可导出
3. 文献清单可导出

## 15. 风险与约束

### 15.1 背景文档过长

风险：

- 每次分析都原样拼接会导致上下文浪费

应对：

- 上传后先解析成结构化摘要
- 后续默认使用结构化摘要

### 15.2 AI 输出太泛

风险：

- 项目总结和对比结果空泛，像模板

应对：

- 强制使用项目背景文档
- 强制引用项目内重点条目
- 输出结构固定

### 15.3 研究助手“越界”

风险：

- 用户误以为 AI 给出的建议已经被验证

应对：

- 全部输出采用辅助性表述
- 保留依据说明
- 鼓励用户继续验证

---

## 16. 结论

`v0.6 研究助手` 的核心不是再增强采集，而是让项目真正拥有持续上下文。

而“项目背景 Markdown 文档”是这一版非常关键的新能力，因为它能把：

- 用户脑海中的问题
- 当前研究方向
- 方法偏好与限制

显式注入系统，成为后续所有 AI 分析的上层上下文。

这将让系统从“理解单条资料”进一步升级为“理解用户当前想研究什么”。
