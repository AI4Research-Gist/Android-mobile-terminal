# Mobile Capture Android Final Spec

状态：最终版  
适用范围：后续安卓端开发必须遵循本规范  
当前实施范围：仅开发安卓端；Web/桌面端本期不实现，只保留数据契约与接口边界

---

## 1. 产品定位

一句话定义：

`手机端负责采集 + 索引补全，电脑端负责整理 + 深度处理。`

这一定义是本规范的最高原则。

对应边界如下：

- 手机端是轻采集入口，不是深度研究工作台
- AI 在手机端只负责补全客观索引信息，不替用户做主观判断
- 分类、状态、处理意图、项目归属由用户自己决定
- 用户自己的想法统一由 `note` 字段承接
- 桌面端负责整理、去重、筛选、阅读、引用、项目管理等重处理工作

---

## 2. 本期目标

本期安卓端需要完成：

- 支持手机端采集论文/资料链接或解析结果
- 自动补全最关键的客观索引字段
- 提供清晰、轻量、低摩擦的详情展示
- 提供显眼的 `note` 输入入口
- 为未来 Web/桌面端同步保留稳定数据结构和接口边界

本期安卓端不做：

- 自动判断“值不值得收”
- 自动生成“待读 / 待引用 / 待整理”等主观状态
- 自动替用户归类到项目或流程阶段
- 生成长摘要
- 生成强解释、强结论、强推荐式内容

---

## 3. 适用对象

本规范主要适用于“论文/资料型采集对象”，包括但不限于：

- 论文链接
- arXiv / DOI 链接
- 会议论文页面
- 学术网页
- 可解析为论文元信息的截图/扫描结果

不直接约束：

- 比赛条目
- 灵感类条目
- 语音类条目

说明：

- 本规范优先落在 Android 当前的“论文采集与详情展示链路”
- 未来若将文章、报告、网页资料也纳入同一采集模型，可以复用本规范的数据字段

---

## 4. 核心产品原则

### 4.1 AI 只做客观索引

AI 在手机端的职责是补全“这是什么”，而不是判断“我要拿它干什么”。

允许 AI 生成：

- 标题
- 作者
- 年份
- 来源
- 标识符
- 领域标签
- 关键词
- 方法标签
- 极短摘要

禁止 AI 自动生成：

- 值不值得收
- 是否重要
- 建议归类
- 阅读优先级
- 后续处理建议
- 引用建议

### 4.2 用户自己负责主观意图

用户的主观输入统一落在：

- `note`

`note` 用于承接：

- 为什么存它
- 它和哪个项目有关
- 自己的瞬时想法
- 之后要在电脑端怎么处理

### 4.3 手机端低摩擦，桌面端重处理

手机端目标：

- 快速保存
- 快速确认
- 快速记一句

桌面端目标：

- 深度整理
- 分类过滤
- 去重归并
- 阅读与引用
- 项目维度管理

---

## 5. 字段优先级

### 5.1 P0 必须字段

以下字段为核心索引字段，安卓端必须优先补全：

- `title`
- `authors`
- `year`
- `venue`
- `identifier`
- `origin_url`

要求：

- 采集成功后优先确保 `P0`
- 允许字段局部缺失，但必须保存已有字段
- `origin_url` 必须保留原始来源，不允许丢失
- `identifier` 优先使用 DOI、arXiv ID 等稳定标识

### 5.2 P1 尽量字段

- `domain_tags`
- `keywords`
- `method_tags`
- `dedup_key`

要求：

- 用于提升检索、去重、桌面端整理效率
- 数量应克制，优先稳定、可检索，不追求花哨
- 安卓端应尽量生成，但允许为空

### 5.3 P2 可选字段

- `summary_short`

要求：

- 只允许 1 到 2 句
- 用于帮助用户快速回忆
- 不替代阅读，不做长摘要
- 若质量不稳定，可为空

### 5.4 User Input

- `note`

要求：

- 安卓端必须提供明显入口
- 用户可在保存前后编辑
- 不做自动改写、不做自动分类

---

## 6. 字段定义

### 6.1 系统解析字段

#### `title`

- 论文或资料标题
- 尽量保留原始正式标题
- 不做润色式改写

#### `authors`

- 用于安卓端展示的作者信息
- 展示层允许做简化，如“第一作者 + 等”

#### `authors_full`

- 完整作者列表
- 用于存储和后续桌面端展示
- 安卓端如当前 UI 不展示完整作者，也应在数据层保留

#### `year`

- 年份
- 优先为发表年份或正式版本年份
- 不确定则可为空

#### `venue`

- 会议 / 期刊 / 来源平台 / 网站来源
- 示例：`NeurIPS`、`ICLR`、`Nature`、`arXiv`

#### `identifier`

- 稳定标识符
- 优先顺序：
  - DOI
  - arXiv ID
  - 其他可稳定识别的外部 ID

#### `origin_url`

- 用户采集时的原始链接
- 必须原样保留

#### `domain_tags`

- 领域标签
- 粗粒度
- 示例：`LLM`、`CV`、`RL`、`HCI`

#### `keywords`

- 检索关键词
- 中粒度
- 建议 3 到 8 个

#### `method_tags`

- 方法类标签
- 示例：`RAG`、`LoRA`、`Diffusion`、`Benchmark`

#### `dedup_key`

- 去重键
- 用于桌面端和服务端归并同一资料
- 应稳定、可重复计算

#### `summary_short`

- 极短摘要
- 最多 1 到 2 句
- 只描述内容，不做评价

### 6.2 用户字段

#### `note`

- 用户备注
- 用于存储主观原因、临时想法和后续动作
- 不参与自动状态判断

## 7. 安卓端数据模型

建议作为统一数据契约保留以下结构：

```json
{
  "id": "string",
  "source_type": "paper",
  "title": "string|null",
  "authors": "string|null",
  "authors_full": ["string"],
  "year": 2025,
  "venue": "string|null",
  "identifier": "string|null",
  "origin_url": "string",
  "domain_tags": ["string"],
  "keywords": ["string"],
  "method_tags": ["string"],
  "dedup_key": "string|null",
  "summary_short": "string|null",
  "note": "string|null",
  "parse_status": "complete|partial|failed",
  "created_at": "ISO-8601",
  "updated_at": "ISO-8601"
}
```

说明：

- 安卓端本期至少要能存储并展示 `P0 + note`
- 若当前数据库结构暂时不能完整承载全部字段，优先保证字段在 `meta_json` 或等价扩展结构中有位置
- 后续正式收敛时，应逐步从松散 JSON 过渡到稳定结构

---

## 8. 安卓端解析规则

### 8.1 解析目标

解析目标优先级如下：

1. 先保证索引完整率
2. 再保证标签可检索性
3. 再考虑极短摘要

不以“摘要写得好不好”作为首要目标。

### 8.2 解析顺序

推荐解析顺序：

1. 提取 `origin_url`
2. 提取 `identifier`
3. 提取 `title`
4. 提取 `authors / authors_full`
5. 提取 `year`
6. 提取 `venue`
7. 生成 `domain_tags`
8. 生成 `keywords`
9. 生成 `method_tags`
10. 生成 `dedup_key`
11. 可选生成 `summary_short`

### 8.3 解析约束

- 不自动判断价值
- 不自动判断处理状态
- 不自动生成项目归属
- 不自动生成阅读优先级
- 不自动生成用户下一步动作
- `summary_short` 不能超过 2 句
- 标签必须短、稳、可检索

### 8.4 `dedup_key` 生成建议

优先级建议：

1. 若存在 DOI，则优先使用 DOI 标准化结果
2. 若存在 arXiv ID，则优先使用 arXiv ID 标准化结果
3. 若无稳定标识，则使用规范化标题 + 年份

规范化要求：

- 小写化
- 去除多余空格
- 去除常见标点差异

---

## 9. 安卓端交互规范

### 9.1 保存流程

建议流程：

1. 用户发起采集
2. 安卓端获得原始内容或链接
3. 执行解析并填充索引字段
4. 展示详情确认页
5. 用户补充 `note`
6. 保存本地并同步后端

### 9.2 详情页展示优先级

详情页首屏优先展示：

- 标题
- 作者
- 年份
- `venue / 来源`
- `identifier`
- `origin_url`
- `domain_tags`
- `keywords`
- `note`

不建议在首屏重点展示：

- 长摘要
- AI 解读
- 推荐理由

### 9.3 `note` 入口要求

`note` 入口必须显眼。

推荐方案：

- 默认展开一个短输入框

可接受方案：

- 明显的“备注”按钮，点击后进入输入态

不建议方案：

- 把备注藏到二级菜单
- 把备注放到很靠后的位置

### 9.4 编辑原则

本期安卓端至少支持：

- 用户编辑 `note`

本期安卓端可选支持：

- 用户手动修正 `title`
- 用户手动修正 `year`
- 用户手动修正 `venue`

但注意：

- 即使允许修正，也不应引入“替用户做分类判断”的逻辑

---

## 10. 安卓端页面要求

### 10.1 列表页

论文/资料列表至少应可展示：

- `title`
- `authors`
- `venue`
- `year`
- 必要时一行 `summary_short`

不要求在列表页展示：

- 长摘要
- 复杂 AI 解读
- 自动状态判断

### 10.2 详情页

详情页至少应包含：

- 标题区
- 作者 / 年份 / 来源区
- 标识符区
- 原始链接区
- 标签区
- 备注区

详情页应支持：

- 查看原始链接
- 复制或查看标识符
- 编辑备注

### 10.3 保存后的状态

保存成功后应确保：

- 本地可立即看到该条记录
- 后续桌面端有能力基于该记录继续整理
- 即使解析不完整，已有字段也不能丢

---

## 11. 存储规范

### 11.1 本期安卓端存储要求

安卓端本期必须保证以下字段可落盘：

- `title`
- `authors`
- `year`
- `venue`
- `identifier`
- `origin_url`
- `note`

建议同时落盘：

- `domain_tags`
- `keywords`
- `method_tags`
- `summary_short`
- `dedup_key`

### 11.2 兼容现有结构的原则

若当前 `ItemEntity` 或 `meta_json` 结构已存在：

- 优先通过向后兼容方式增加字段
- 不做高风险重构
- 安卓端以“先能保存、先能展示、先能同步”为第一目标

### 11.3 机器字段与用户字段分离

存储层必须能明确区分：

- 系统解析字段
- 用户手动字段

即使暂时都放在同一条记录内，也要保证语义层面清晰分离。

---

## 12. Web / 桌面端预留契约

本期不实现 Web/桌面端，但安卓端开发必须为其预留接口边界。

### 12.1 数据契约要求

安卓端同步出的数据必须至少包含：

- `title`
- `authors_full`
- `year`
- `venue`
- `identifier`
- `origin_url`
- `domain_tags`
- `keywords`
- `method_tags`
- `dedup_key`
- `summary_short`
- `note`

### 12.2 Web / 桌面端未来职责

未来 Web/桌面端负责：

- 去重归并
- 按标签搜索
- 按项目/流程归类
- 阅读状态管理
- 引用意图管理
- 深度摘要和深度处理

### 12.3 预留接口边界

未来接口建议以以下能力为边界：

- 创建采集记录
- 更新 `note`
- 获取记录详情
- 基于 `identifier / dedup_key` 去重查询
- 基于标签字段搜索

安卓端当前实现时应避免把这些能力硬编码成“仅手机端自用”的私有逻辑。

---

## 13. 本期安卓实施范围

本期安卓端必须实现：

1. 采集后保留 `origin_url`
2. 自动补全 `P0` 字段
3. 尽量补全 `domain_tags / keywords`
4. 详情页展示索引信息
5. 提供显眼的 `note` 输入和编辑入口
6. 保存本地数据
7. 为 Web/桌面端同步保留完整数据字段位置

本期安卓端可选实现：

1. `method_tags`
2. `summary_short`
3. 更完整的 `dedup_key`
4. 手动修正部分客观字段

本期安卓端明确不做：

1. 自动推荐
2. 自动值判断
3. 自动流程状态判断
4. 自动项目归属
5. 长摘要
6. 强论文解读
7. Web 页面或桌面整理能力

---

## 14. 验收指标

本规范采用以下指标验收：

### 14.1 索引完整率

检查以下字段是否完整：

- `title`
- `authors`
- `year`
- `venue`
- `identifier`

### 14.2 关键词可用率

检查以下字段是否足以支持后续检索：

- `domain_tags`
- `keywords`
- `method_tags`

### 14.3 去重成功率

检查是否能稳定识别同一论文：

- `identifier`
- `dedup_key`

### 14.4 人工补录率

检查用户到桌面端后还需补多少客观索引字段。

### 14.5 备注使用率

检查用户是否愿意在手机端补充自己的主观意图。

不再作为主指标：

- 长摘要质量
- 强解释质量
- 自动推荐准确率

---

## 15. MVP 验收清单

达到以下条件即可认为安卓端第一阶段达标：

- 用户能在手机端完成一次论文/资料采集
- 系统能保存 `origin_url`
- 系统能补全大部分 `P0` 字段
- 详情页能清晰展示索引信息
- 用户能方便地填写和修改 `note`
- 数据结构已为未来 Web/桌面端预留字段位置

---

## 16. 开发约束

从本文件开始，后续开发必须遵循以下约束：

- 安卓端优先实现采集、索引补全、备注输入
- 不把手机端做成深度研究处理工具
- 机器字段和用户字段必须分离
- 所有新增功能都必须先判断是否符合“手机端轻采集、桌面端重处理”的原则
- 若未来需求与本规范冲突，以本规范为准，除非先更新本规范

当前默认模型实现说明：

- Provider：`SiliconFlow`
- 文本模型：`Qwen/Qwen3.5-397B-A17B`
- 视觉模型：`Pro/moonshotai/Kimi-K2.5`
- ASR 模型：`FunAudioLLM/SenseVoiceSmall`

说明：

- 当前项目不是直接接入 Qwen 官方 SDK
- 当前项目也不是直接接入 Moonshot / Kimi 官方 SDK
- 所有模型调用仍通过 SiliconFlow API 统一转发
- 如需查看模型调用量和日志，应优先查看 SiliconFlow 控制台

---

## 17. 最终结论

本规范确定了产品的最终方向：

- 手机端负责采集
- AI 负责索引补全
- 用户自己决定主观意图
- `note` 承接用户想法
- 桌面端负责整理与深度处理

后续安卓开发，按本规范执行。Web/桌面端当前只保留接口与字段契约，不进入本期实现范围。
