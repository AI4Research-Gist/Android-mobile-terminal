## 2. 需求分析与立项依据

### 2.1 现状与痛点

在科研调研与竞赛备赛中，“信息获取”越来越移动端化，但“知识沉淀”仍停留在 PC 端或手工流程。典型日常是真实且高频的：同学在地铁、课间刷公众号，在浏览器看 arXiv 或博客，在群聊里接收转发链接，或者从截图中看到关键信息，当下能读，但后续却很难变成可复用资产。

这种断裂主要表现为四类痛点，它们往往同时发生：

1. 信息过载，内容看得多但筛不动。
2. 入口分散，来源杂且难统一管理。
3. 整理低效，从“看见”到“变成可用资料”需要多次跳转。
4. 沉淀薄弱，多数收藏最终只留下链接，无法形成知识卡片。

进一步展开，这些痛点体现在：

- 当天“看过/收藏”很多，但一周后几乎回忆不起。
- 需要写综述、做方案或准备比赛材料时，找不到当时为什么收藏、关键点是什么。
- 收藏位置分散在微信、浏览器、群聊、相册、PDF 阅读器等不同入口。
- 同一内容反复出现、反复搜索、反复整理。
- 标题、作者、年份、出处、关键要点经常需要手动二次确认。
- 比赛通知还需要自行抽取时间节点、任务要求和准备清单。

---

### 2.2 需求与市场证据

上述痛点不是个别用户习惯问题，而是由外部环境共同驱动的系统性需求，至少有三类证据支撑：

1. 移动端已经成为科研信息消费与传播的事实入口。
2. 科研与竞赛相关内容持续增长，筛选与整理成本不断抬高。
3. 信息过载已有明确研究结论，会导致筛选困难与决策质量下降。

这意味着问题的关键不只是“信息太多”，而是“信息进入系统后没有被结构化”。  
因此项目需要解决的不是简单收藏，而是让用户在移动端完成“抓住主旨、留住要点、后续可复用调用”的最小闭环。

可在正式汇报材料中补充一张需求证据图：

- 移动端入口趋势
- 内容供给增长趋势
- 信息过载研究结论

共同指向：**结构化沉淀需求**。

---

### 2.3 竞品与差距

现有替代方案并不少，但在“移动端入口 -> 自动结构化 -> 可复用沉淀”这条完整链路上，仍有明显缺口：

1. 微信收藏、浏览器书签、转发到电脑这类方式入口强，但沉淀弱。
2. Zotero 等文献管理器强在引用与库管理，但弱在移动端高频剪藏和非论文内容结构化。
3. 通用笔记与通用 AI 总结工具能生成内容，但难形成可追踪、可重试、可验收的工程闭环。

这些方案的共性问题是：

- 往往只能保存链接，缺少摘要、要点、标签、启发等结构化字段。
- 对公众号、网页、比赛通知、碎片灵感等非标准论文内容支持不足。
- 缺少稳定模板、状态管理、失败重试、结果审计等工程要素。

因此，本项目的立项逻辑可以明确归纳为：

**现有工具要么入口强但不结构化，要么结构化依赖手工或不稳定，要么缺少工程闭环。Gist 的目标就是以移动端为入口，把“收藏”升级为“结构化知识资产”，并保证结果可复用、过程可控。**

---

## 3. 一句话定义测试版前的项目

在进入测试版之前，**Gist** 已经从一个以安卓端采集为核心的 AI 工具，演进为一个面向科研/学习场景的 **移动端个人 AI 研究助手**：

- 支持多入口采集：链接、截图、图片、语音、手动灵感
- 支持统一条目管理：`paper / article / competition / insight / voice`
- 支持 AI 解析与阅读辅助：摘要、OCR、阅读卡、条目问答
- 支持知识连接：论文去重、资料关联论文、灵感关联条目
- 支持项目级能力：项目总览、背景 Markdown、AI 项目总结
- 支持输出与复盘：双文献对比、Markdown 导出、研究回顾、同步诊断

测试版前的核心结论是：

**产品闭环已经成立，测试版的任务更多是集成收口与材料整理，而不是从零定义产品。**

## 5. 分阶段开发脉络

### 5.1 孵化期：从 Android 工程到可演示原型

时间范围：

- `2025-12-18` 到 `2026-02-02`

代表提交：

- `Initial commit: AI4Research Android project`
- `Add comprehensive README.md`
- `ui优化，功能完善`
- `feat: 完善AI4Research项目功能 - 修复Gradle缓存问题并添加前台服务权限`
- 多次 `uidesign / function update / voice update / LOGO修改`

这一阶段的重点不是版本化产品，而是做出一个“能跑、能看、能继续加能力”的原型底座。  
可以把这一段定义为：

**项目形态与技术方向确认期。**

阶段产出包括：

- Android 应用工程正式建立
- 产品最早期 UI 风格和图标资源成型
- 语音入口、基础功能、权限模型、前台服务等雏形逐步出现
- README 和工程说明初步建立

这一阶段的价值在于：虽然还没有明确的版本号，但已经完成了产品方向和工程方向的“第一次定型”。

---

### 5.2 `v0.0.x`：把项目从原型拉到可演示状态

`v0.0.2 ~ v0.0.5` 的主线不是 AI 深化，而是把基本运行条件和比赛展示能力补齐。

这个阶段依次解决了四件事：

1. **远端可用**：`v0.0.2` 完成 NocoDB 迁移与配置统一  
2. **数据隔离**：`v0.0.3` 完成比赛版用户分离  
3. **采集入口可用**：`v0.0.4` 修复底部 `+` 号入口  
4. **比赛条目能管理**：`v0.0.5` 完成比赛时间线与排序逻辑

这一阶段的项目定位，仍然更接近“研究场景采集器”，但已经有了三项很关键的正式产品意识：

- 账号隔离
- 真实数据链路
- 面向比赛和答辩的展示稳定性

---

### 5.3 `v0.1.x`：从采集工具迈向结构化论文助手

`v0.1.0 / v0.1.1 / v0.1.5` 是第一轮真正有“研究工具产品感”的阶段。

这个阶段的核心动作是：

- 给论文条目补齐结构化索引字段
- 建立中英文摘要与切换机制
- 引入“资料”作为新的内容类型，为公众号、网页、碎片信息留入口

这意味着产品从一开始偏单一的论文解析工具，开始扩展成：

**既能接论文，也能接研究资料的统一条目系统。**

这一阶段奠定了后续多个版本都在持续复用的基础：

- `meta_json` 作为结构化信息承载层
- 论文与资料共用的条目管理思路
- 摘要、索引、笔记、详情展示的统一框架

---

### 5.4 `v0.2.x`：把 OCR、链接、图片导入做成正式链路

`v0.2.0 ~ v0.2.3` 是采集链路真正产品化的阶段。

它解决的不是“AI 能不能解析”，而是“真实用户输入东西之后，系统能不能稳定接住”。

阶段关键变化如下：

- `v0.2.0`：先把 OCR 截图识别链路修稳
- `v0.2.1`：链接导入改为先分类入库、后台解析、状态可视化
- `v0.2.2`：资料页改成本地优先显示，不再等远端成功才出现
- `v0.2.3`：相册图片导入、多页 OCR 整理、AI 整理与入库链路正式化

到这个阶段结束时，Gist 已经具备比较完整的“多入口采集系统”：

- 复制链接可导入
- 截图可 OCR
- 图片可整理
- 内容可先落本地再逐步完善

这对后续版本非常关键，因为 `v0.4+` 的阅读辅助和 `v0.5+` 的知识连接，都建立在“内容先稳定进系统”之上。

---

### 5.5 `v0.3.x`：灵感模块正式独立

`v0.3.0 ~ v0.3.2` 是产品结构上的一次明显重构。

这一阶段不再把首页当作泛聚合页，而是直接把首页收敛为“灵感库”。  
这代表项目开始承认一个事实：

**研究过程里，灵感不是附属品，而是需要被单独管理的一类核心内容。**

阶段关键变化：

- `v0.3.0`：首页改成灵感页，支持手动录入、详情查看、图片与原始语音展示
- `v0.3.1`：灵感从自由标签收敛到归位词条模型
- `v0.3.2`：去重分类、摘要展示收口、解析输出容错增强

这一阶段之后，项目中的五类核心条目已经全部比较明确：

- `paper`
- `article`
- `competition`
- `insight`
- `voice`

也就是说，到 `v0.3.x` 结束时，Gist 的内容版图已经基本成型。

---

### 5.6 `v0.4.0`：AI 首次进入阅读过程

`v0.4.0` 是测试版前最重要的转折点之一。

在这之前，AI 主要负责：

- 链接解析
- OCR 整理
- 摘要生成

到了 `v0.4.0`，AI 开始参与“内容已经进系统以后，用户怎么继续读”：

- 为论文 / 资料生成结构化阅读卡
- 围绕当前条目进行小模型问答
- 把阅读卡回写到条目元数据中

这意味着产品不再只是采集整理工具，而开始具备“阅读型研究助手”的特征。

---

### 5.7 `v0.5.x`：从单条内容进入知识连接

`v0.5.0 ~ v0.5.1` 的核心贡献，是把条目之间真正连起来。

这里的变化不是表面 UI，而是系统理解信息组织方式发生了升级：

- 同一论文支持去重分组
- `article -> paper` 支持自动关联
- `insight` 支持手动关联已有条目
- 项目从“筛选维度”升级为“可浏览总览”

`v0.5.1` 则进一步把底座做稳：

- 新增同步诊断
- 数据库升级切换为正式 migration
- 项目背景 Markdown 上传、保存、摘要开始落地

这一阶段的意义是：

**Gist 不再只是“很多卡片放在一起”，而是开始形成研究知识网络。**

---

### 5.8 `v0.6.x`：项目级研究助手正式成形

`v0.6.0 ~ v0.6.2` 是测试版前的最后一个阶段，也是最接近完整产品形态的阶段。

三个版本分工很清晰：

- `v0.6.0`：项目级 AI 能力正式出现
- `v0.6.1`：结果支持导出、回顾与复盘
- `v0.6.2`：同步问题收口、补传能力上线

这个阶段的关键新增能力包括：

- 项目总览页生成 `AI 项目总结`
- 灵感详情页 `反查相关资料`
- 论文 / 资料详情页 `双文献对比`
- 项目总结与对比结果 `导出 Markdown`
- 设置页 `研究回顾`
- 设置页 `同步诊断`
- 未上云条目 `立即补传`

到 `v0.6.2` 为止，项目已经具备一个测试版产品应有的核心形态：

- 能采
- 能存
- 能看
- 能问
- 能连
- 能围绕项目总结
- 能导出
- 能诊断和补救同步问题

## 6. 由需求分析衍生出的系统功能分解与结构设计

这一章用于回答一个更“开发文档化”的问题：从前面的需求分析出发，项目实际是如何把需求拆解成可实现的系统模块、层次结构、调用关系、模块接口和人机界面的。

### 6.1 需求分析结果到功能目标的映射

基于第 2 章中的痛点分析，项目将需求映射为 5 类核心功能目标：

1. 针对“信息过载”，需要提供结构化摘要、阅读卡、AI 问答与项目总结能力。
2. 针对“入口分散”，需要提供链接、截图、图片、语音和手动灵感的统一采集入口。
3. 针对“整理低效”，需要提供自动抽取标题、作者、年份、标签、时间线、摘要等结构化字段。
4. 针对“沉淀薄弱”，需要提供项目归属、条目关系、知识连接、导出与复盘能力。
5. 针对“过程不可控”，需要提供本地优先、同步诊断、失败补传与状态可视化能力。

也就是说，需求分析并没有直接落成“几个页面”，而是被转换成一套完整的能力目标链：

`多入口采集 -> 结构化入库 -> 统一管理 -> AI 辅助阅读 -> 知识连接 -> 项目级总结与复盘 -> 同步诊断与补救`

---

### 6.2 功能模块分解

测试版前系统已经可以拆解为以下 8 个核心功能模块：

1. 用户与认证模块  
负责注册、登录、会话保持、用户隔离与测试账号归档。

2. 多入口采集模块  
负责链接导入、悬浮窗采集、截图 OCR、相册导入、语音录制、手动灵感创建。

3. AI 解析与结构化模块  
负责链接解析、OCR 整理、双语摘要、阅读卡生成、条目问答、项目总结、文献对比等 AI 能力。

4. 条目管理模块  
负责 `paper / article / competition / insight / voice` 五类条目的统一存储、列表展示、筛选、详情查看和编辑。

5. 知识连接模块

### 6.4 模块调用关系

系统的调用关系总体上遵循“界面发起 -> ViewModel 编排 -> Repository/Service 执行 -> 本地与远端数据落地 -> 结果回传界面”的模式。

几个典型调用链如下：

1. 链接采集链路

`用户输入链接 / 悬浮窗检测链接 -> JS Bridge / FloatingWindowService -> MainViewModel -> AIService 解析 -> ItemRepository 入库 -> Room 更新 -> WebView 列表刷新`

2. OCR / 图片导入链路

`用户截图或选择图片 -> Image/OCR Service -> AIService 整理 -> ItemRepository 创建条目 -> Room 保存 -> 异步同步 NocoDB -> 详情页/列表页可见`

3. 条目阅读链路

`用户打开详情页 -> DetailViewModel -> ItemRepository 读取条目 -> DetailScreen 展示 -> AIService 生成阅读卡或回答问题 -> DetailViewModel 更新状态 -> 用户保存回 meta_json`

4. 项目级总结链路

`用户进入项目总览 -> ProjectOverviewViewModel 汇总项目条目 -> AIService 生成项目总结 -> ProjectOverviewScreen 展示 -> 用户导出 Markdown`

5. 同步补传链路

`用户在设置页点击立即补传 -> MainViewModel.retryUnsyncedItems() -> ItemRepository.syncLocalItemToRemote() -> NocoDB -> 更新诊断结果 -> WebView 设置页刷新`

从结构上看，系统的调用关系不是点状的，而是围绕“统一条目模型”展开的网状组织。

## 8. 数据库设计说明

### 8.1 数据库总体说明

测试版前本地数据库采用 Room，数据库名为 `ai4research_db`，版本号为 `6`。  
数据库承担本地单一事实源（SSOT）角色，所有主页面展示优先依赖本地库，再由远端异步同步补齐。

测试版前已纳入数据库的核心实体包括：

- `items`
- `projects`
- `users`
- `item_relations`
- `project_context_documents`

---

### 8.2 核心表结构

1. `items`

用于存储统一研究条目，关键字段包括：

- `id`
- `owner_user_id`
- `type`
- `title`
- `summary`
- `content_md`
- `origin_url`
- `audio_url`
- `status`
- `read_status`
- `project_id`
- `project_name`
- `is_starred`
- `meta_json`
- `created_at`
- `synced_at`

2. `projects`

用于存储项目主信息，关键字段包括：

- `id`
- `owner_user_id`
- `name`
- `description`
- `created_at`

3. `users`

用于缓存当前登录用户信息，关键字段包括：

- `id`
- `email`
- `username`
- `phone`
- `avatarUrl`
- `biometricEnabled`
- `updatedAt`

4. `item_relations`

用于存储条目之间的关系，关键字段包括：

- `id`
- `owner_user_id`
- `from_item_id`
- `to_item_id`
- `relation_type`
- `confidence`
- `source`
- `created_at`

5. `project_context_documents`

用于存储项目背景 Markdown 文档及其摘要信息，关键字段包括：

- `id`
- `owner_user_id`
- `project_id`
- `title`
- `markdown_path`
- `summary`
- `keywords`
- `updated_at`

---

### 8.3 数据关系与约束

测试版前，本地数据关系可概括为：

- 一个 `user` 拥有多条 `item`
- 一个 `user` 拥有多个 `project`
- 一条 `item` 可归属一个 `project`
- 一条 `item` 可通过 `item_relations` 与其他条目建立多种关系
- 一个 `project` 最多对应一份 `project_context_document`

其中较重要的约束包括：

- `item_relations` 对 `(owner_user_id, from_item_id, to_item_id, relation_type)` 建立唯一索引
- `project_context_documents` 对 `(owner_user_id, project_id)` 建立唯一索引
- 所有核心业务数据都显式带有 `owner_user_id`，用于用户隔离

---

### 8.4 数据库演进策略

测试版前数据库已明确切换到正式 migration 策略，不再依赖 destructive migration。

当前已知迁移包括：

- `4 -> 5`：新增 `item_relations`
- `5 -> 6`：新增 `project_context_documents`

测试版前的数据库策略可以总结为：

- 优先增量演进
- 保留已有用户数据
- 升级路径可验证
- 不以“清库”作为常规升级手段

## 13. 测试版前已经稳定形成的产品基线

在写任何后续材料时，以下表述可以视为**测试版前已经成立的基线事实**。

### 13.1 产品定位基线

Gist 不是泛聊天产品，也不是单纯的论文摘要器。  
测试版前更准确的定义应当是：

**面向科研/学习场景的移动端个人 AI 研究助手。**

关键词应优先使用：

- 研究条目管理
- 多入口采集
- 结构化阅读
- 知识连接
- 项目级研究助手

不建议把它简单写成：

- AI 聊天软件
- 论文翻译工具
- 纯笔记软件

---

### 13.2 技术架构基线

测试版前已经稳定形成的架构口径如下：

- **端侧**：Android，Hybrid UI（WebView + Compose）
- **本地数据层**：Room 作为 SSOT
- **远端数据层**：NocoDB
- **AI 能力层**：SiliconFlow / Qwen 系列
- **应用架构**：MVVM + Repository + Hilt
- **关键链路特点**：本地优先、异步同步、渐进式完善

这套架构不是临时实验，而是已经支撑多个版本持续演进的正式技术基线。

---

### 13.3 内容模型基线

测试版前的核心条目类型已经确定为五类：

- `paper`
- `article`
- `competition`
- `insight`
- `voice`

辅助组织单元包括：

- `project`
- `item relations`
- `reading_card`
- `project background markdown`

后续材料如果描述功能范围，应优先围绕这套内容模型展开。
