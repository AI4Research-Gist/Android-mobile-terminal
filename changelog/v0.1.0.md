# v0.1.0 - 2026-03-19

## 版本主题
安卓端“论文采集 + 索引补全”第一阶段落地版。

本版本重点不是增加桌面端或 Web 端功能，而是把安卓端采集后的数据结构、索引字段、备注字段和展示行为先稳定下来，为后续 Web / 桌面端开发提供一致的数据契约。

---

## 本次版本完成内容

### 1. 论文索引字段扩展
安卓端当前已经支持在 `meta_json` 中承载并消费以下字段：

- `authors`
- `conference`
- `year`
- `source`
- `identifier`
- `domain_tags`
- `keywords`
- `method_tags`
- `dedup_key`
- `summary_short`
- `note`

### 2. 用户备注字段独立
新增并落地：

- `note`

设计原则：

- AI 负责客观索引
- 用户主观输入统一进入 `note`
- `note` 不与 `summary` 混用

### 3. 安卓详情页展示升级
论文详情页当前已经支持单独展示：

- 作者
- 来源 / venue
- 年份
- identifier
- 原始链接
- 极短摘要 / Preprint 总结
- 领域标签
- 关键词
- 方法标签
- 备注

### 4. 列表卡片索引化
安卓首页论文卡片当前优先展示：

- 论文标题
- 作者
- 来源 / 年份
- identifier
- 极短摘要
- 备注
- 标签

### 5. 解析链路增强
安卓端当前解析链路包含：

1. 主结构化抽取
2. 规则兜底
3. 缺字段时二次索引补全

当前规则兜底包括：

- `arXiv ID -> year`
- 标题关键词兜底
- 领域标签弱推断
- 论文类型自动归类优化

### 6. 兼容远端 `meta_json` 字符串
修复并兼容以下两种远端返回格式：

1. `meta_json` 为 JSON 对象
2. `meta_json` 为 JSON 字符串

这样 Web / 后端在联调时不必强依赖单一格式。

## 面向 Web 端开发者的开发手册

## 1. 当前产品边界

当前版本的职责划分是：

- 手机端负责采集、索引补全、轻量展示、备注输入
- Web / 桌面端负责整理、过滤、深度处理、归类、引用等重操作

因此 Web 端在接入时，不应假设安卓端已经完成：

- 自动项目归类
- 自动主观优先级判断
- 自动工作流状态判断
- 完整研究工作台能力

安卓端本期只负责把“可继续整理的数据”稳定产出。

---

## 2. 当前推荐数据模型

面向 Web / 桌面端，当前应按以下语义理解一条论文记录：

```json
{
  "id": "string",
  "type": "paper",
  "title": "string",
  "summary": "string",
  "content_md": "string",
  "origin_url": "string",
  "status": "processing|done|failed",
  "read_status": "unread|reading|read",
  "project_id": "string|null",
  "project_name": "string|null",
  "meta_json": {
    "authors": ["string"],
    "conference": "string|null",
    "year": "string|null",
    "source": "string|null",
    "identifier": "string|null",
    "domain_tags": ["string"],
    "keywords": ["string"],
    "method_tags": ["string"],
    "dedup_key": "string|null",
    "summary_short": "string|null",
    "note": "string|null"
  }
}
```

说明：

- `summary` 是兼容字段，可能是较长摘要或主摘要
- `summary_short` 是移动端优先展示的极短摘要
- `note` 是用户主观输入
- Web 端后续应优先消费 `meta_json` 中的结构化字段，而不是只依赖 `summary`

---

## 3. 字段语义说明

### `title`
论文标题，优先使用客观原始标题。

### `authors`
作者列表。
安卓端展示层可能会只显示第一作者或部分作者，但存储层建议消费完整结构。

### `conference`
会议、期刊、venue 或来源。
如果没有明确 venue，安卓端可能回退为来源类型或平台信息。

### `year`
年份。
可能来自：

- AI 抽取
- 规则推断
- arXiv ID fallback

### `source`
来源类型或来源平台，如：

- `arxiv`
- `doi`
- `wechat`
- `web`

### `identifier`
稳定标识符，优先级高于标题匹配。
常见值：

- DOI
- arXiv ID

### `domain_tags`
粗粒度学术领域标签。
例如：

- `LLM`
- `NLP`
- `CV`

### `keywords`
检索关键词。
安卓端列表展示当前优先使用这一组字段。

### `method_tags`
偏方法论标签。
例如：

- `RAG`
- `LoRA`
- `Transformer`

### `dedup_key`
去重键。
当前安卓端会尽量生成，但仍建议 Web / 后端保留自己的去重策略。

### `summary_short`
极短摘要。
目标是移动端快速理解，不是完整学术摘要。

### `note`
用户备注。
这个字段代表用户主观意图，不应被 AI 输出覆盖。

---

## 4. 安卓端当前行为说明

### 4.1 详情页显示逻辑
当一条记录被识别为 `PAPER` 时：

- 优先展示 `PaperMeta`
- 优先展示 `summary_short`
- 若 `summary_short` 缺失，则回退到 `summary`

### 4.2 列表卡片显示逻辑
论文列表卡片当前优先显示：

- `summary_short`
- 若没有，则回退到 `summary`

关键词标签显示优先级：

1. `keywords`
2. `domain_tags`
3. `method_tags`
4. 旧 `tags`

### 4.3 远端返回兼容逻辑
安卓端当前兼容：

- `meta_json` 为对象
- `meta_json` 为字符串形式的 JSON

如果 Web / 后端后续能统一只返回对象，会更稳定，但当前不是硬要求。

### 4.4 论文类型自动归类
当解析结果带有学术特征时，即使模型返回：

- `article`
- `insight`

安卓端也可能自动归类为 `PAPER`。

判定依据包括：

- `identifier`
- `conference`
- `year`
- `arxiv`
- `doi`

## 5. Web 端接入建议

### 5.1 不要只读 `summary`
如果 Web 端继续只读旧的：

- `title`
- `summary`

那么会丢掉本次版本最重要的结构化索引能力。

建议优先读取：

- `meta_json.identifier`
- `meta_json.conference`
- `meta_json.year`
- `meta_json.domain_tags`
- `meta_json.keywords`
- `meta_json.method_tags`
- `meta_json.summary_short`
- `meta_json.note`

### 5.2 `note` 与 AI 字段必须分开
后续 Web 端开发时应保持：

- AI 字段是客观索引
- `note` 是用户主观输入

不要把 `note` 当成摘要，也不要被后续补全逻辑覆盖。

### 5.3 去重逻辑建议
Web / 后端后续去重建议优先级：

1. `identifier`
2. `dedup_key`
3. 规范化 `title + year`

安卓端当前只提供辅助，不应作为唯一去重来源。

---

## 6. 当前已知边界

### 6.1 年份和关键词不是强确定字段
当前即使升级模型，以下字段依然可能缺失：

- `year`
- `keywords`
- `domain_tags`
- `method_tags`

原因：

- 网页本身未明确提供
- 抓取内容不完整
- 这些字段本身更偏推断而不是纯抽取

因此 Web 端在消费时需要接受“部分为空”的现实，不应把这些字段当成强必填。

### 6.2 `summary_short` 更适合入口展示
`summary_short` 适合：

- 列表页
- 卡片页
- 移动端入口展示

不适合替代：

- 原始英文摘要
- 完整 Markdown 内容

### 6.3 安卓端仍然是单端优先
当前版本没有完成：

- Web UI
- 桌面整理工作台
- 旧记录批量重建索引

Web 端如果要继续推进，需要按当前数据契约继续做增量实现。
