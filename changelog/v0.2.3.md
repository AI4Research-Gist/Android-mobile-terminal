# v0.2.3 - 2026-03-23

## 版本主题
Android 端“图片扫描导入 + 多页 OCR 整理 + 敏感配置脱敏”升级版本。

本版本相对 `v0.2.2` 的核心变化，不再只是修补资料条目的本地优先显示，而是把“图片采集 -> OCR -> AI 整理 -> 条目入库 -> 远端同步”这一条链路正式做成了可复用能力，同时补齐了截图授权稳定性、文章元数据展示以及构建层的敏感配置脱敏。

---

## 本次版本完成内容

### 1. 主界面新增“从相册导入图片”扫描入口
本轮在主界面中把原先的“扫描”动作升级为系统图片选择流程：

- 点击扫描后，不再直接走悬浮窗截图，而是支持从系统相册一次选择多张图片。
- 选择图片后，会先弹出导入配置弹窗，让用户选择目标分类：
  - `资料`
  - `论文`
  - `竞赛`
  - `动态`
- 同时支持在导入前直接选择项目归属，未分配项目也可直接导入。
- 条目加入队列后，前端会自动切到对应 Tab，并给出“正在后台整理”的提示。

这一步让“手动导入已有截图 / 相册图片”正式成为和悬浮窗链接导入并列的采集入口，而不再局限于实时截图。

### 2. 新增统一图片扫描导入服务 `ImageScanImportService`
本轮新增独立服务 `ImageScanImportService`，把图片导入能力从 UI 和悬浮窗逻辑中抽离出来，统一负责：

- 将图片 `Uri` 拷贝为本地可处理文件。
- 创建本地 `processing` 占位条目。
- 调用 OCR 与 AI 整理链路。
- 将整理后的标题、摘要、正文、标签、元数据回填到本地条目。
- 最后再异步同步到 NocoDB。

具体能力包括：

- `queueImport(...)`
  - 负责先把扫描任务排入本地条目队列。
- `processQueuedImport(...)`
  - 负责后台继续执行 OCR、整理、回填和同步。
- `importFromUris(...)`
  - 支持从系统选择器导入多张图片。
- `importFromLocalPaths(...)`
  - 支持从截图本地路径直接导入。

这让“相册导入”和“截图导入”开始共用同一套扫描整理链路，后续扩展成本明显下降。

### 3. OCR 能力从“识别正文”升级为“结构化识别 + 多页整理”
`AIService` 本轮是变化最大的模块之一，OCR 能力从单张图片文本识别，升级为更完整的结构化整理流程：

- `recognizeImageStructured(...)` 现在不只返回 `title/authors/doi/content`，还会补充：
  - `identifier`
  - `identifierType`
  - `referencedLinks`
- 新增 `organizeOcrBatch(...)`
  - 支持把多张截图识别结果合并整理为一个统一条目。
- 新增 OCR 批处理提示词：
  - 自动识别 `paper/article/competition/insight`
  - 提取链接、DOI、arXiv 编号、账号名、标签、候选论文线索
  - 输出中英摘要、短摘要、核心点、主题标签等结构化信息
- 新增多种兜底逻辑：
  - 从 OCR 原文中直接抽取 DOI / arXiv
  - 从原文中提取链接并规范化为 `https://doi.org/...` 或 `https://arxiv.org/abs/...`
  - 当 AI 结构化失败时，仍能生成保底条目，而不是整条链路直接失败

这意味着扫描图片导入已经不是“存一张图 + 一段 OCR 文本”，而是接近“扫描后直接生成可用资料卡片”的体验。

### 4. 链接、论文编号与来源判断能力增强
在 `AIService` 中，本轮还补齐了很多“输入不标准但仍应识别成功”的场景：

- 支持直接输入裸 DOI 或裸 arXiv ID，并自动补全为标准链接。
- URL 解析前会先做标准化，避免因为输入格式不完整导致抓取失败。
- `guessContentTypeFromUrl(...)` 与 `guessSourceFromUrl(...)` 增强了 DOI / arXiv 判定。
- `extractIdentifierFromUrl(...)` 先尝试通用 DOI / arXiv 提取，再回退到旧规则。

这部分改动的结果是：

- 输入研究编号时，不必强制要求用户先粘贴完整链接。
- OCR 中出现的论文编号，也更容易被整理成可跳转的论文线索。

### 5. 文章元数据结构增强，详情页展示更完整
本轮对 `ArticleMeta` 相关结构做了增强，开始正式保留和展示“文章里提到的论文线索”：

- `ResearchItem.ArticleMeta` 新增 `identifier` 字段。
- `ItemMapper` 新增对 `identifier` 的双向映射。
- `MainViewModel` 导出的 JSON 结构中，文章元数据新增 `identifier`。
- `DetailScreen` 的资料信息卡中新增展示：
  - 提及编号
  - 提取到的链接列表

这让资料条目不再只是“一个标题 + 摘要”，而是开始具备“可追溯到论文线索”的结构化索引价值。

### 6. 条目创建与更新链路补齐项目归属和来源链接
本轮对 `ItemRepository` 做了多项接口扩展，解决图片扫描条目后续落库时的信息丢失问题：

- `createFullItem(...)` 与 `createLocalPendingItem(...)` 新增：
  - `projectId`
  - `projectName`
- 本地占位条目和远端正式条目都支持保留项目归属。
- `updateItem(...)` 新增 `originUrl` 参数。
- 更新条目时增强了标签与 `meta_json` 的合并逻辑，避免局部回填时把已有元数据覆盖掉。

这保证了扫描导入条目在“创建 -> 解析 -> 回填 -> 远端同步”过程中，不会丢失项目归属、原始链接和标签信息。

### 7. 截图导入改为复用统一扫描链路
悬浮窗截图完成后的处理逻辑，本轮已经切换到新的统一导入服务：

- `FloatingWindowService` 新增 `handleScreenshotV2(...)`
- 截图完成后不再走旧的单点处理逻辑，而是直接调用 `ImageScanImportService.importFromLocalPaths(...)`
- 成功时直接展示整理结果
- 失败时仍会保底创建本地图片条目，避免截图成果丢失

这一步的意义在于：

- 截图和相册图片终于进入同一条处理管线
- OCR、整理、失败兜底策略得到统一
- 后续维护不再需要同时改两套“图片入库”逻辑

### 8. 截图权限与 MediaProjection 会话稳定性增强
本轮补了一组偏底层、但对截图成功率很关键的修复：

- `MediaProjectionStore` 新增：
  - Android 14+ 兼容的复用策略判断
  - 最近一次投影错误信息记录
  - `prepareProjection(...)`
  - `peekLastProjectionError()`
- `ScreenCaptureActivity` 重构截图启动流程：
  - 获取授权后，不再直接把动作转回 Service，而是先在 Activity 内部建立稳定会话
  - 若投影会话创建失败，会给出更明确的错误提示并重新请求授权
  - Android 14+ 使用 `MediaProjectionConfig.createConfigForDefaultDisplay()`
- 清理流程改为通过 `MediaProjectionStore.releaseProjection()` 控制，避免投影对象被不恰当地复用或提前失效

这组改动主要面向两个问题：

- 部分设备截图授权拿到了，但后续无法真正建立投影会话
- 新版本 Android 下旧的授权复用方式不稳定

### 9. SiliconFlow 视觉模型与 OCR 提示词同步调整
本轮在 `SiliconFlowApiService` 中把视觉模型配置切换为与当前文本模型同一套 `Qwen/Qwen3.5-397B-A17B`，并在 `AIService` 中重写了 OCR 提示词：

- 提示词更强调严格 JSON 输出
- 更强调 DOI / arXiv / 链接提取
- 更强调按证据保守归类

这一调整与多页整理能力一起，属于整套扫描导入升级的模型侧配套改动。

### 10. API Key / Token 脱敏，改为本地环境配置
本轮把项目中的敏感配置从源码硬编码迁移到本地环境文件 / 系统环境变量：

- `app/build.gradle.kts` 新增本地 `.env.local` 读取逻辑
- 新增 BuildConfig 注入字段：
  - `NOCO_BASE_URL`
  - `NOCO_TOKEN`
  - `SILICONFLOW_BASE_URL`
  - `SILICONFLOW_API_KEY`
- `Constants.kt` 改为从 `BuildConfig` 读取 NocoDB 配置
- `AIService.kt` 改为从 `BuildConfig.SILICONFLOW_API_KEY` 生成认证头
- `SiliconFlowApiService.kt` 改为从 `BuildConfig.SILICONFLOW_BASE_URL` 读取地址
- `.gitignore` 新增忽略 `.env.local`
- 新增 `.env.local.example`
- `README.md`、`Gist.md`、`CLOUD_WEB_API.md` 同步更新为新的配置方式

这一步的目标很明确：

- 避免 API Key / Token 继续明文写在 Kotlin 源码中
- 让本地开发、后续 CI 注入和多人协作都更安全

## 本轮相对 v0.2.2 解决的问题

### 修复前
- 图片扫描主要仍偏单次截图处理，缺少统一导入服务。
- 无法从主界面一次选择多张图片并按分类 / 项目导入。
- OCR 结果偏原始，难以直接形成结构化资料条目。
- 文章条目中提到的 DOI / arXiv / 链接线索缺少统一保存与展示。
- 截图授权在部分设备或 Android 新版本上存在会话不稳定问题。
- API Key / Token 仍然存在源码硬编码风险。

### 修复后
- 相册导入、多页扫描、截图导入都开始走统一服务。
- 图片可先进入本地占位条目，再后台 OCR 与 AI 整理。
- 多页 OCR 能生成结构化条目、链接线索、论文候选与摘要。
- 资料详情页能直接展示提及编号和提取到的链接。
- 截图会话建立失败时有更明确的恢复与提示逻辑。
- 敏感配置迁移到 `.env.local` / 系统环境变量，源码层完成脱敏。

## 当前版本边界

本版本已经把“图片扫描导入”做成了正式链路，但仍有一些边界值得后续继续观察：

- 多页 OCR 的最终质量仍依赖截图清晰度与页面完整度。
- 扫描导入失败后虽然会保留占位条目，但重试机制仍可继续增强。
- `network_security_config.xml` 仍保留对当前 NocoDB 主机的明文 HTTP 放行，这解决的是连通性，不属于本轮脱敏范围。
- `.env.local` 只适用于本地开发环境，正式构建仍建议接入 CI 级环境变量注入。

## 版本结论

`v0.2.3` 相比 `v0.2.2`，不只是补一个局部 Bug，而是把“图片扫描导入”从边缘能力推进成了正式能力：

- 用户现在可以从主界面直接选择多张图片导入
- OCR 结果开始具备结构化整理与论文线索提取能力
- 截图与相册导入开始走统一链路
- 资料条目的可读性与可追溯性明显增强
- 敏感配置完成了从源码硬编码到本地环境注入的迁移

如果说 `v0.2.2` 解决的是“资料条目要先看得见”，那么 `v0.2.3` 解决的就是“图片采集进来之后，能够被更稳定、更结构化、更安全地整理成条目”。
