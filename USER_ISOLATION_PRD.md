# Gist 扩展 PRD：用户分离能力（比赛落地版）

## 1. 背景

当前项目已经具备注册、登录、条目管理、项目管理等基础能力，但数据层仍是“弱多用户”状态：

- `users` 表存在，但 `items` 和 `projects` 没有用户归属字段
- Android 刷新数据时直接全量拉取 `projects` 和 `items`
- 本地 Room 也没有按用户隔离数据
- 登录后保存的“token”本质上只是 `user.id`

这意味着当前系统在业务上并没有真正实现“一个账号只看到自己的数据”。

对外表现为：
- A 用户登录后，理论上可能拿到 B 用户创建的条目/项目
- 同一设备切换账号时，本地缓存存在串号风险
- 云端 Web 若直接接入当前 NocoDB 表，也难以保证数据隔离

因此需要补齐“用户分离”能力，作为后续正式上线、云端 Web 接入、多人使用的基础能力。

## 2. PRD 目标

本 PRD 以“计算机设计大赛可展示、可答辩、可落地”为目标，不追求一步到位做成完整权限系统。

本次扩展要实现的核心目标：

1. 每个用户只能看到和操作自己的数据
2. 同设备切换账号时，不出现明显串数据
3. 在答辩时可以清楚说明产品已具备基础多用户隔离能力
4. 改动范围控制在当前项目可承受的工作量内
5. 为后续正式认证体系升级预留扩展空间

## 3. 非目标

比赛阶段本次明确不做：

- 完整 RBAC 权限系统
- JWT / 刷新令牌 / 风控体系等完整认证基础设施
- 团队协作 / 项目共享 / 多人共编
- 企业级 IAM / SSO
- 全量替换 NocoDB 为自建后端
- 为历史脏数据设计复杂自动修复流程

本次只解决：

- 私有数据隔离
- 单用户数据边界
- 演示和答辩场景下的数据安全感与产品完整度

## 4. 当前问题定义

### 4.1 云端问题

当前 `items` / `projects` 表没有 `user_id`：
- 服务端无法按用户过滤数据
- 查询接口只能返回全量或靠前端自行筛

### 4.2 客户端问题

Android 当前行为：
- `refreshProjects()` 拉全量项目
- `refreshItems()` 拉全量条目
- Room 本地表也不区分归属用户
- 登出时虽然会清 `users`，但未来若支持更复杂缓存，风险会放大

### 4.3 认证问题

当前登录方式是：
- 直接查 `users` 表
- 用 `user.id` 当“token”存本地

问题：
- 不是真正认证票据
- 难以承载服务端鉴权
- 不利于 Web / Android / 云端统一接入

## 5. 用户故事

### 5.1 普通用户

作为注册用户，我希望：
- 登录后只能看到自己的论文、比赛、灵感、语音卡片
- 新建的项目和条目默认归属于我
- 切换账号后不会看到上一个账号的数据

### 5.2 Web 管理 / 云端开发

作为云端 / Web 开发者，我希望：
- 所有业务数据都有清晰的 `owner` 字段
- 接口天然按用户过滤，不需要前端再做数据清洗
- 认证信息可以稳定传递给后端用于鉴权

### 5.3 产品 / 运维

作为产品或运维，我希望：
- 用户投诉“串号”时能快速定位归属
- 数据迁移有明确策略
- 后续要做团队共享时，现有模型能平滑升级

## 6. 需求范围

本次用户分离覆盖以下对象：

1. 用户
2. 条目 `items`
3. 项目 `projects`
4. 本地缓存
5. 登录态 / 鉴权信息

比赛阶段范围控制说明：

- 必做：`items`、`projects`、登录后数据拉取、创建数据写入归属、切换账号隔离
- 可选：本地缓存做“按用户过滤”或“切账号清缓存后重拉”，二选一即可
- 暂缓：复杂权限管理、管理员视角、共享数据模型

## 7. 功能方案总览

### 7.1 比赛推荐方案

推荐采用“最小可用用户分离”方案：

1. 云端数据层
   - 给 `items` 和 `projects` 增加 `owner_user_id`
   - 所有新建数据都写入当前用户归属

2. 查询层
   - 所有列表查询只查当前用户的数据
   - 不再使用全量拉取再本地筛选的方式

3. Android 本地层
   - 方案优先级 A：Room 增加 `ownerUserId`，所有查询按用户过滤
   - 方案优先级 B：若时间紧，切换账号时直接清缓存并重新同步

### 7.2 为什么这样定

原因：

- 这已经足够解决比赛中最容易被问到的“多账号数据是否隔离”问题
- 改动成本远低于重做完整认证体系
- 对现有架构侵入相对可控
- 可以作为后续正式产品化的第一步

## 8. 数据模型设计

### 8.1 云端表结构调整

#### users

保留现有表，建议补充：

| 字段 | 类型 | 说明 |
|---|---|---|
| `Id` | Int / UUID | 用户唯一标识 |
| `email` | String | 邮箱 |
| `username` | String | 用户名 |
| `password_hash` | String | 推荐替代明文密码 |
| `status` | String | 用户状态 |

说明：
- 若短期仍使用 NocoDB 原表，可先保留 `password`，中期迁移为 `password_hash`

#### items

新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `owner_user_id` | Int / String | 所属用户 ID，必填 |

实现备注：
- PRD 语义字段统一写作 `owner_user_id`
- 当前比赛环境中的 NocoDB 实际物理字段名为 `ownerId`
- Android 本轮实现会直接对接现网 `ownerId`

可选新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `created_by` | Int / String | 创建者 |
| `updated_by` | Int / String | 最后修改者 |

#### projects

新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `owner_user_id` | Int / String | 所属用户 ID，必填 |

说明：
- 若未来需要共享项目，可继续扩展 `project_members`
- 本阶段先以单 owner 为准

### 8.2 Android Room 调整

#### ItemEntity

新增字段：

```kotlin
@ColumnInfo(name = "owner_user_id")
val ownerUserId: String
```

#### ProjectEntity

新增字段：

```kotlin
@ColumnInfo(name = "owner_user_id")
val ownerUserId: String
```

#### 查询要求

所有本地查询都要带 `ownerUserId` 过滤：

- `observeAllItemsByUser(ownerUserId)`
- `observeItemsByTypeAndUser(ownerUserId, type)`
- `observeProjectsByUser(ownerUserId)`

## 9. 认证与鉴权设计

## 9.1 长期目标状态

长期推荐目标：

- 登录成功后由服务端签发真正的访问凭证
- 客户端保存 `access_token`
- 所有业务请求由服务端解析 token 得到当前用户

### 推荐接口

```http
POST /auth/register
POST /auth/login
POST /auth/logout
GET /auth/me
```

### 推荐返回

```json
{
  "access_token": "xxx",
  "user": {
    "id": "123",
    "email": "a@example.com",
    "username": "alice"
  }
}
```

## 9.2 比赛阶段过渡方案

如果比赛阶段继续使用 NocoDB 直连，最低要求：

1. 登录后客户端至少同时持久化：
   - `current_user_id`
   - `current_user_email`

2. 所有业务写入必须显式带上：
   - `owner_user_id = current_user_id`

3. 所有读取必须显式加 where：
   - `where=(owner_user_id,eq,{current_user_id})`

说明：
- 这不是最终推荐方案
- 但它非常适合作为比赛版上线方案
- 先保证“能隔离、能演示、能说明”，后续再升级正式认证

## 10. 接口需求

## 10.1 items

### 查询我的 items

```http
GET /items
```

服务端行为：
- 根据当前登录用户自动过滤 `owner_user_id`

如果仍走 NocoDB 直连过渡方案，则 Android 侧请求应改为：

```http
GET /mez4qicxcudfwnc?where=(owner_user_id,eq,123)&limit=100&offset=0&sort=-CreatedAt
```

### 创建 item

创建时应自动写入：

```json
{
  "title": "xxx",
  "type": "paper",
  "owner_user_id": "123"
}
```

要求：
- 前端不可伪造为其他用户
- 若走正式后端，应由服务端从 token 注入 owner

### 更新 / 删除 item

要求：
- 只能操作自己名下数据
- 非 owner 返回无权限错误

## 10.2 projects

### 查询我的 projects

```http
GET /projects
```

要求：
- 仅返回 `owner_user_id = current_user_id`

### 创建 project

要求：
- 自动绑定当前用户为 owner

### 删除 project

要求：
- 仅 owner 可删

可选策略：
- 删除项目时是否联动清除 item.project_id
- 推荐保留 item，只把其 `project_id` 置空

## 10.3 auth

### 注册

要求：
- 用户名、邮箱唯一
- 返回标准 user 信息

### 登录

要求：
- 返回真实 token，而不是只返回 user id

### 获取当前用户

```http
GET /auth/me
```

用途：
- Android 启动恢复会话
- Web 刷新页面恢复登录状态

## 11. Android 端改造需求

### 11.1 TokenManager

新增或明确保存：

- `access_token`
- `current_user_id`
- `current_user_email`

不再把 `user.id` 当作 token 直接混用。

### 11.2 AuthRepository

改造目标：

1. 登录后写入真实会话信息
2. `getCurrentUser()` 以 `current_user_id` 为主
3. 登出时清理当前会话和对应本地数据视图

### 11.3 ItemRepository

改造目标：

1. `refreshItems()` 必须带用户上下文
2. 创建 item 时自动绑定当前用户
3. 读取 Room 时按当前用户过滤

### 11.4 ProjectRepository

改造目标：

1. `refreshProjects()` 必须带用户上下文
2. 创建 project 时自动绑定当前用户
3. 本地查询按当前用户过滤

### 11.5 MainViewModel / DetailViewModel

改造目标：

- 不需要感知复杂鉴权细节
- 只消费“已经被当前用户过滤过”的数据流

### 11.6 本地迁移策略

Room 版本需要升级。

迁移建议：

1. 方案 A：破坏式迁移
   - 开发阶段最快
   - 会清掉本地旧缓存

2. 方案 B：正式迁移脚本
   - 给历史本地数据打上当前登录用户 owner
   - 更复杂，但体验更好

推荐：
- 比赛版优先使用破坏式迁移或切账号清缓存方案
- 不在这一阶段投入过多精力做复杂迁移

## 12. 云端 / Web 端改造需求

### 12.1 Web 列表页

要求：
- 默认只拉当前用户数据
- 不允许通过修改前端参数看到其他用户数据

### 12.2 Web 新建与编辑

要求：
- 创建时不暴露 owner 字段给用户选择
- owner 由登录态确定

### 12.3 Web 管理后台

如果后续有管理端：
- 普通用户视角只看自己
- 管理员视角另开权限，不与普通接口混用

## 13. 迁移策略

### 13.1 存量云端数据

需要给历史 `items/projects` 补 owner。

可选策略：

1. 基于创建人日志回填
2. 基于现有测试账号人工归类
3. 无法识别的历史数据统一归到内部管理员账号

推荐：
- 先梳理当前线上/测试环境使用者
- 做一次性迁移脚本

### 13.1.a 比赛版固定策略

为降低比赛阶段迁移复杂度，采用固定规则：

1. 新增一个“测试归档账号”
2. 当前已有的历史 `items` 和 `projects`，统一迁移到该测试归档账号
3. 后续新注册用户默认从空数据开始
4. 比赛演示时，如需展示完整历史内容，统一使用该测试归档账号登录

这样做的好处：

- 不需要为历史测试数据逐条人工认领
- 不会因为旧数据没有 owner 而阻塞用户分离改造
- 比赛演示账号的数据完整、稳定、可控
- 普通用户账号天然具备“只看到自己数据”的效果

### 13.1.b 测试归档账号规范

建议固定一个内部测试账号，专门承接迁移前的历史数据：

| 项目 | 建议值 |
|---|---|
| 用途 | 承接历史数据 / 比赛演示 / 回归验证 |
| 账号类型 | 内部测试账号 |
| 用户名 | `gist_demo_archive` |
| 邮箱 | `gist_demo_archive@example.com` |
| 数据范围 | 迁移前已有的全部历史 items / projects |

约束：

1. 该账号不作为正式用户运营账号
2. 该账号密码不写入公开仓库文档
3. 该账号仅由项目组内部掌握，用于比赛演示和迁移兜底
4. 若比赛前整理出更干净的演示数据，可继续只保留该账号下的数据

### 13.1.c 迁移执行规则

迁移时执行以下规则：

1. 若 `items.owner_user_id` 为空，则统一回填为测试归档账号 ID
2. 若 `projects.owner_user_id` 为空，则统一回填为测试归档账号 ID
3. 迁移完成后，所有新建 item / project 都必须强制写入当前登录用户 ID
4. 迁移完成后，不再允许出现 owner 为空的数据

### 13.1.d 比赛答辩表述建议

比赛中可以这样说明：

- “为了保证多用户隔离，我们将历史测试数据统一归档到内部测试账号。”
- “普通新用户登录后不会看到历史公共测试数据。”
- “这样既保留了演示素材，也保证了正式多用户隔离逻辑成立。”

### 13.2 存量本地数据

可接受策略：
- 升级版本后本地清库重新同步

适用前提：
- 当前产品仍处于开发 / 内测期

## 14. 风险与约束

### 14.1 最大风险

1. 历史数据没有 owner，迁移成本不可避免
2. 当前登录体系不规范，继续直连 NocoDB 的安全性有限
3. Android 本地多账号切换容易遗漏某些查询过滤

### 14.2 技术债

当前系统已有以下技术债会影响实现：

1. 登录 token 语义不清
2. 客户端直连 NocoDB
3. `users.password` 仍是明文使用
4. `starred` 目前只存在本地，不在云端

## 15. 成功标准

### 功能验收标准

1. 用户 A 登录后，只能看到 A 的 items / projects
2. 用户 B 登录后，只能看到 B 的 items / projects
3. A 和 B 在同一设备切换登录，不会串数据
4. 创建 / 更新 / 删除接口对非 owner 拒绝
5. Android 刷新时不再拉全量公共数据

### 比赛答辩标准

在比赛展示和答辩中，产品应能明确表达：

1. 系统支持基础多用户隔离
2. 用户登录后只可访问自己的研究条目和项目
3. 新建数据自动归属到当前账号
4. 切换账号后不会直接看到上一个账号的数据

### 数据验收标准

1. 云端新增数据 `owner_user_id` 填充率 100%
2. 迁移后历史有效数据 owner 归属明确
3. 用户投诉“看到别人数据”的问题为 0

## 16. 推荐实施阶段

### Phase 1：比赛 MVP

目标：
- 快速实现“每个用户只看到自己的数据”

范围：
- 云端给 `items/projects` 增加 `owner_user_id`
- Android 所有刷新请求按当前用户过滤
- 创建数据自动绑定 owner
- 本地采用“按用户过滤”或“切账号清缓存重拉”二选一

不做：
- 正式 token 体系
- 完整后端网关
- 共享权限模型

### Phase 2：产品化增强

目标：
- 把比赛版方案升级为更稳的正式方案

范围：
- Room 全量按 `ownerUserId` 隔离
- 上线 `/auth/login`、`/auth/me`
- 客户端改存 `access_token`
- 服务端从 token 自动识别用户

### Phase 3：协作能力预留

目标：
- 为团队协作做模型扩展

范围：
- `project_members`
- `shared_items`
- 读写权限模型

2. 给 `projects` 增加 `owner_user_id`
3. 登录后查询只查当前用户数据

1. Room 增加 `ownerUserId`

3. 增加“当前账号”展示，提升演示可信度

### 后端 / 数据

1. 给 `items` 增加 `owner_user_id`
2. 给 `projects` 增加 `owner_user_id`
3. 编写历史数据迁移脚本
4. 新增按 owner 过滤的查询接口

### Android

4. `refreshItems/refreshProjects` 增加用户上下文
5. 创建 item / project 自动写 owner
6. 登出 / 切账号流程校验本地数据隔离

### Web

1. 所有列表页改为只读当前用户数据
2. 新建 / 编辑流程接入当前用户上下文
3. 补充无权限态处理

## 18. 实施建议

作为比赛产品，“用户分离”值得做，但只建议做比赛 MVP，不建议现在就把项目拖进完整权限体系。

推荐顺序：

1. 先用 `owner_user_id` 把 `items/projects` 的归属补齐
2. 再把查询改成“只查当前用户”
3. 然后处理切账号时的本地缓存隔离
4. 正式认证体系留到比赛后或下一阶段

这样做的收益是：

- 能显著提升产品完整度
- 能回答评委对多用户场景的质疑
- 实现成本相对可控
- 不会因为后端体系过重影响比赛节奏

如果现在开工，建议直接按 Phase 1 执行。
