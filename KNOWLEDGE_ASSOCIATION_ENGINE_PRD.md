# Knowledge Association Engine PRD

## 1. 文档定位

本文档定义 `Gist / AI4Research` 后续拟开发的“知识联想引擎”产品方案。

该能力的核心目标不是直接替用户“生成创新点”，而是：

- 将历史论文 / 文章沉淀为可复用的研究记忆
- 在新条目导入时自动召回相关历史知识
- 基于关联结果给出“潜在研究机会 / 可能的组合方向 / 可验证假设”
- 帮助用户更快完成科研资料整理、相关工作联想与灵感产生

本文档主要面向：

- 产品设计
- Android 端实现
- AI 工作流设计
- 后续数据结构扩展

---

## 2. 背景与问题

当前项目已经具备以下基础能力：

- 多入口信息采集：链接、截图 OCR、语音、灵感记录
- AI 解析与摘要
- 本地数据存储与项目归属
- 条目详情页查看、编辑、AI 对话

当前能力的优势在于“采集”和“单条整理”，但在以下环节仍存在明显空缺：

- 历史条目之间缺乏真正的知识关联
- 新导入论文无法自动利用历史阅读积累
- AI 问答主要围绕单条内容展开，尚未形成项目级研究记忆
- 灵感与论文/文章之间缺乏系统化联想链路

因此，需要新增一层“知识联想引擎”，把单条资料管理升级为研究记忆系统。

---

## 3. 产品目标

### 3.1 核心目标

1. 让每条论文 / article 在导入后都生成一份“中等规模总结 + 结构化知识卡”
2. 让这些知识卡可被机器检索，也可被人阅读
3. 在新条目导入时自动召回相关历史条目
4. 使用本地小模型完成粗筛与关系判断
5. 基于候选条目输出“潜在研究机会”而非空泛建议

### 3.2 非目标

本阶段不追求：

- 全自动高可信创新结论生成
- 替代用户完成文献综述写作
- 完整的向量数据库或大规模知识图谱
- 多人协作知识库
- 复杂云端同步体系

---

## 4. 产品原则

### 4.1 研究辅助，不冒充研究结论

系统输出的文案应统一使用以下术语：

- 潜在研究机会
- 可能的组合方向
- 可验证假设
- 关联启发

避免直接使用：

- 创新点已生成
- 最佳方案
- 结论已确定

### 4.2 双存储原则

知识沉淀不只依赖 Markdown。

- 结构化存储负责：检索、召回、排序、拼接上下文
- Markdown 存储负责：人读、浏览、导出、项目知识镜像

### 4.3 模型分层原则

- 大模型负责高质量压缩和高层综合生成
- 小模型负责低成本路由、粗筛、打标、关系分类

### 4.4 逐步增强原则

第一版优先实现：

- 知识卡生成
- Markdown 镜像
- 规则召回
- 小模型粗判断

不在第一版直接做复杂语义检索或向量库。

## 5. 用户故事

### 5.1 作为科研用户

当我导入一篇新论文后，我希望系统不仅给我一个摘要，还能告诉我：

- 这篇论文和我之前收过的哪些内容相关
- 它可能和哪些方法、任务、数据集产生结合
- 值不值得我进一步深入看

### 5.2 作为长期使用者

当我积累了很多论文和文章后，我希望这些历史内容不会只是静态卡片，而是会在新资料导入时参与推理，形成研究记忆。

### 5.3 作为项目型用户

当我在某个项目下持续收集资料时，我希望系统能围绕这个项目形成项目级知识库，而不是一堆彼此孤立的条目。

## 6. 目标用户与适用场景

### 6.1 目标用户

- 科研学生
- AI / 计算机方向研究者
- 比赛型用户
- 做长期项目追踪与资料整理的学习者

### 6.2 适用场景

- 新论文导入后的快速定位
- article 中提到论文后的补充联想
- 某一项目下不同资料的组合探索
- 灵感与历史知识结合
- 准备开题 / 综述 / related work 前的材料梳理

## 7. 整体工作流

### 7.1 导入后知识沉淀

新论文 / article 导入成功后：

1. 解析原始内容
2. 使用较大模型生成：
   - 中等规模总结 `medium_summary`
   - 结构化知识卡
   - 潜在启发种子 `innovation_seeds`
3. 结果写入结构化存储
4. 同步写入项目 Markdown 知识库

### 7.2 新条目联想流程

当新条目完成基础解析后：

1. 从历史条目中召回候选
2. 用本地小模型对候选做粗判断
3. 输出每个候选与新条目的关系类型和分数
4. 筛出高分候选
5. 将：
   - 新条目知识卡
   - 高分候选知识卡
   - 小模型关系判断
   一起交给较强模型
6. 生成潜在研究机会
7. 展示给用户，并允许保存为灵感

## 9. 数据设计

### 9.1 结构化知识卡字段

建议统一放在条目 `meta_json` 中：

- `medium_summary`
- `research_question`
- `method`
- `dataset`
- `findings`
- `limitations`
- `innovation_seeds`
- `related_topics`
- `retrieval_keywords`
- `knowledge_status`
- `last_synthesized_at`

### 9.2 字段说明

#### `medium_summary`

用于：

- 给用户展示一个比 `summary_short` 更完整的内容
- 作为后续候选召回和上下文拼接的主要摘要文本

#### `research_question`

表示条目解决的核心问题。

#### `method`

表示方法、框架、范式、策略。

#### `dataset`

表示显式提到的数据集、评测集或实验对象。

#### `findings`

表示主要结论或核心观察。

#### `limitations`

表示局限、约束、前提条件。

#### `innovation_seeds`

表示初步的联想方向，不是最终创新点，可为数组。

#### `related_topics`

表示主题归属，用于项目内聚类。

#### `retrieval_keywords`

表示给召回用的关键词，优先精炼和可搜索。

#### `knowledge_status`

可选值建议：

- `raw`
- `summarized`
- `linked`
- `synthesized`

#### `last_synthesized_at`

用于追踪知识卡最近一次生成时间。

## 10. Markdown 知识库设计

### 10.1 目标

Markdown 知识库不是主数据源，而是项目级“知识镜像”。

用途：

- 人工浏览
- 项目导出
- 项目级二次喂模
- 长期沉淀与复查

### 10.2 推荐目录结构

```text
knowledge/
  projects/
    {projectId}/
      index.md
      papers/
        {itemId}.md
      articles/
        {itemId}.md
      synthesis/
        {yyyyMMdd}-{itemId}.md
```

### 10.3 单条知识卡 Markdown 模板

```md
# {Title}

## Type
paper / article

## Medium Summary
...

## Research Question
...

## Method
...

## Dataset
...

## Findings
...

## Limitations
...

## Innovation Seeds
- ...
- ...

## Retrieval Keywords
- ...
- ...

## Source
- itemId: ...
- projectId: ...
- originUrl: ...
```

### 10.4 项目 index.md 模板

```md
# Project Knowledge Index

## Topics
- Topic A
- Topic B

## Recent Papers
- [Title A](./papers/{itemId}.md)

## Recent Articles
- [Title B](./articles/{itemId}.md)

## Recent Syntheses
- [Synthesis 1](./synthesis/{date}-{itemId}.md)
```

## 11. 候选召回策略

### 11.1 第一层：规则召回

第一版不引入向量库，使用以下规则召回：

- 同项目优先
- 相同 `domain_tags`
- 相同 `retrieval_keywords`
- 相同或相近 `method`
- 相同或相近 `dataset`
- title / medium_summary 命中关键词
- article 的 `paperCandidates` 中命中已有 paper

### 11.2 第二层：排序加权

候选排序可参考：

- 同项目加权
- 星标加权
- 已读且有阅读卡加权
- 最近导入加权
- 有 `innovation_seeds` 加权

### 11.3 候选数建议

- 初始召回：`top 10`
- 小模型粗筛后保留：`top 3 ~ 5`

## 12. 模型工作流设计

### 12.1 较大模型职责

用于导入后沉淀与综合联想。

职责：

- 生成中等规模总结
- 生成结构化知识卡
- 提取创新种子
- 对高分候选进行综合分析
- 生成潜在研究机会

### 12.2 本地小模型职责

用于低成本快速判断。

职责：

- 判断历史条目是否与新条目相关
- 分类关系类型
- 进行候选粗排序
- 给出是否值得进入下一轮大模型分析

### 12.3 不建议让小模型直接负责

- 产出高可信研究结论
- 独立生成完整创新方案
- 替代大模型完成复杂综合分析

---

## 13. 关系分类设计

本地小模型第一版建议输出固定关系标签：

- `same_problem`
- `same_method`
- `method_transfer`
- `dataset_transfer`
- `cross_domain_analogy`
- `background_only`
- `probably_irrelevant`

同时输出一个粗分数：

- `0.0 ~ 1.0`

示例：

```json
{
  "candidate_id": "xxx",
  "relation_type": "method_transfer",
  "score": 0.78,
  "reason": "两者任务不同，但都依赖多模态对齐和检索增强"
}
```

## 14. 综合输出设计

### 14.1 输出名称

前端展示建议统一用：

- 潜在研究机会
- 组合方向
- 可验证假设

### 14.2 大模型输出格式

建议结构化为：

- `opportunity`
- `why_it_matches`
- `possible_combination`
- `validation_path`
- `risk_or_counterpoint`

示例：

```json
{
  "opportunity": "将论文A中的检索增强机制迁移到论文B的多模态推理任务中",
  "why_it_matches": "两者都依赖外部知识补充，但一个强调召回，一个强调推理",
  "possible_combination": "使用A的检索模块为B提供候选证据，再由B的推理模块做选择",
  "validation_path": "先在小规模公开 benchmark 上做 ablation",
  "risk_or_counterpoint": "A 的检索策略可能不适用于 B 的长上下文结构"
}
```

## 19. 风险与约束

### 19.1 模型幻觉风险

风险：

- 将“可能的组合方向”误表达为可靠结论

应对：

- 文案层面统一使用启发式表述
- 明确输出 `risk_or_counterpoint`

### 19.2 Markdown 不适合作为唯一真相

风险：

- 后续检索、召回、更新变得混乱

应对：

- 结构化层始终为主
- Markdown 仅作为镜像

### 19.3 小模型能力边界

风险：

- 小模型输出粗糙、误判高

应对：

- 小模型只负责初筛与标签
- 高价值输出仍交给更强模型

### 19.4 性能与成本

风险：

- 每次导入都触发大模型，成本较高

应对：

- 导入时只做一次高质量沉淀
- 联想分析时先小模型粗筛，再决定是否调用大模型

## 22. 结论

“知识联想引擎”将把当前项目从：

- 采集工具
- 单条摘要器
- 本地资料库

升级为：

- 研究记忆系统
- 跨条目关联助手
- 项目级知识沉淀工具
- 潜在研究机会发现工具

这一能力会显著提升项目的差异化，也更符合科研工具长期价值。
